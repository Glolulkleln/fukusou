 服装租赁小程序研发模块清单）
### 一、用户端（C端小程序）模块明细

| 序号 | 主模块名称 | 核心功能点明细 | 研发避坑指南 |
| :--- | :--- | :--- | :--- |
| 1 | **登录与授权** | 1. 微信授权登录（获取头像、昵称）。<br>2. 手机号快捷授权（用于订单联系和物流通知）。<br>3. 无需实名认证，无需学号验证，所有微信用户均可使用。 | **注意：** 由于无实名认证，需在租赁协议中明确用户对账号行为的责任，通过收取押金对冲风险。 |
| 2 | **首页展示** | 1. 搜索框（按名称、场景搜索）。<br>2. 轮播图（发布毕业季、新品活动）。<br>3. 场景化分类（面试正装、社团演出、汉服写真等）。<br>4. 推荐/上新列表（支持下拉加载）。<br>5. 服装规格自定义（不同尺码可设不同价格和库存）。 | **设计铁律：** 42.71%的用户认为功能模块清晰是决定使用的首要因素。 |
| 3 | **服装详情与租赁** | 1. 多图展示（实拍图+细节+上身效果）。<br>2. 规格与租期选择（按天、按预设套餐）。<br>3. 立即租赁与购物车（支持多件合并下单）。<br>4. 押金与须知（明确标注押金金额、损坏赔偿标准、归还流程）。 | **信任核心：** 拒绝过度修图，展示真实瑕疵，建立用户信任。 |
| 4 | **订单与归还** | 1. 全流程状态跟踪（待支付→租赁中→待归还→已完成）。<br>2. 快递物流跟踪（展示快递单号和物流状态）。<br>3. 自动消息提醒（归还到期前24小时发送模板消息）。 | **体验红线：** 租赁流程压缩至3-4步内完成。 |
| 5 | **押金管理** | 1. 下单时随租金一并支付押金。<br>2. 归还确认后自动发起退款，24小时内退还。<br>3. 押金扣除明细记录（如有损坏赔偿）。 | **转化核心：** 押金退还要快，这是复购的关键。 |
| 6 | **个人中心** | 1. 用户信息管理。<br>2. 我的订单（全部/待支付/租赁中/已归还）。<br>3. 我的收藏。<br>4. 收货地址管理。<br>5. 联系客服。 | **必要项：** 必须包含联系客服入口。 |
| 7 | **评价与买家秀** | 1. 订单完成后48小时内可评价（星级+图文）。<br>2. 精选买家秀展示（在服装详情页展示真实上身图）。 | **护城河：** 真实评价是建立信任壁垒的核心武器。 |


### 二、管理后台（PC端）模块明细

| 序号 | 主模块名称 | 核心功能点明细 | 研发侧重点 |
| :--- | :--- | :--- | :--- |
| 1 | **服装管理** | 增删改查、上下架、多规格库存管理、分类维护。 | 需支持一键上架新品，方便与服装厂对接。 |
| 2 | **订单管理** | 全渠道订单筛选、异常订单处理、手动确认归还。 | 必须支持“确认归还后自动触发押金退还”流程。 |
| 3 | **用户管理** | 用户列表、租赁记录、黑名单禁用。 | 建立黑名单机制（基于手机号或微信号）。 |
| 4 | **财务管理** | 押金流水对账、收入统计报表、退款审核。 | 所有资金操作需留痕（日志），确保账实相符。 |
| 5 | **数据看板** | 核心指标（订单量/用户数）、热门排行、校区分布占比。 | 通过数据看板指导精准选品和运营决策。 |
| 6 | **权限管理** | 超级管理员（全权）/ 校区运营（本校区）/ 仓库（仅入库）。 | 代理与管理员权限分离，防止核心数据泄露。 |


### 三、数据库核心表结构设计（建表参考）

| 核心数据表 | 关键字段设计 | 逻辑说明 |
| :--- | :--- | :--- |
| **用户表** | id, 微信昵称, 头像, 手机号, 注册时间, 状态(正常/黑名单)。 | 以手机号作为用户唯一标识。 |
| **服装表** | id, 名称, 分类ID, 主图, 多规格JSON（如S/M/L对应的库存和价格）, 租金/天, 押金金额, 状态(上架/下架), 创建时间。 | 将规格存为JSON，可无限扩展尺码，无需改表结构。 |
| **分类表** | id, 分类名称, 排序值。 | 用于首页场景化分类展示。 |
| **订单表** | id, 订单号(唯一), 用户ID, 服装ID, 规格选择, 租赁起止时间, 实际归还时间, 租金金额, 押金金额, 实付总额, 订单状态(待支付/租赁中/待归还/已完成/已取消), 快递单号, 创建时间。 | 根据“租赁起止时间”和“实际归还时间”自动计算是否逾期。 |
| **押金流水表** | id, 关联订单号, 用户ID, 金额, 类型(支付/退还/扣除), 状态, 备注, 操作时间。 | 每笔押金必须有明细，避免纠纷。 |
| **评价表** | id, 订单号, 用户ID, 服装ID, 评分(1-5星), 评价内容, 评价图片, 创建时间。 | 关联服装表，详情页展示评价。 |
| **轮播图表** | id, 图片地址, 跳转链接, 排序值, 状态(启用/停用)。 | 首页活动配置。 |
| **管理员表** | id, 用户名, 密码哈希, 角色(超级管理员/校区运营/仓库), 手机号, 状态。 | 分角色权限控制。 |


### 四、技术栈推荐（零成本自研方案）

| 层级 | 推荐技术 | 说明 |
| :--- | :--- | :--- |
| 小程序端 | 微信原生开发（WXML+WXSS+JS）或 UniApp | 原生稳定；UniApp可跨端，根据团队熟悉度选择。 |
| 后端 | SpringBoot（Java）或 Node.js | 任一熟悉的技术栈即可。 |
| 数据库 | MySQL（阿里云RDS或自建） | 开源免费，成熟稳定。 |
| 缓存 | Redis | 用于热点数据缓存和防超售。 |
| 支付 | 微信支付 | 需企业主体资质（入驻学校创业园可办理）。 |
| 短信通知 | 腾讯云/阿里云短信服务 | 用于订单状态通知。 |
| 云服务器 | 阿里云/腾讯云学生版或初创企业套餐 | 前期免费额度完全够用。 |


### 五、支付与押金流程设计

| 环节 | 操作内容 | 说明 |
| :--- | :--- | :--- |
| **下单支付** | 用户选择服装→选择租期→系统自动计算租金+押金→微信支付一次性付清。 | 租金和押金合并支付，简化流程。 |
| **归还确认** | 用户寄回服装→仓库收到后检查确认→管理员在后台点击“确认归还”。 | 管理员手动确认，确保检查无误。 |
| **押金退还** | 确认归还后→系统自动发起押金退款→原路退回用户微信账户。 | 需确保24小时内到账。 |
| **损坏处理** | 检查发现损坏→与用户沟通确认→后台扣除相应金额→剩余押金退还。 | 需保留损坏照片作为凭证。 |


### 六、核心业务流程图

**用户操作流程：**

微信授权登录 → 浏览服装（分类/搜索/首页推荐） → 进入详情页（查看图片/规格/评价） → 选择规格和租期 → 加入购物车或立即租赁 → 填写收货地址 → 支付（租金+押金合并支付） → 订单生成（状态：待发货） → 管理员发货 → 用户收到服装（状态：租赁中） → 到期前24小时收到归还提醒 → 用户寄回服装 → 管理员收到并检查确认（状态：待归还→已完成） → 24小时内押金退还 → 邀请用户评价

**管理员操作流程：**

登录后台 → 收到新订单 → 拣货、打包、发货 → 标记“已发货” → 收到归还包裹 → 检查服装（完好/损坏） → 确认归还 → 系统自动退还押金 → 处理异常（损坏扣款/逾期扣款）
